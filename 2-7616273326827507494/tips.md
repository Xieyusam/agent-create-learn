## 本次项目复盘补充

以下问题在后续项目中要默认规避：

### 1. 密钥与配置管理

- 不要在提示词、`config.yaml`、README、脚本或测试样例中硬编码真实 API Key。
- 必须提供 `.env.example`，真实密钥只放 `.env` 或系统环境变量。
- `config.yaml` 中涉及密钥的字段，优先使用 `${ENV_NAME}` 占位符，例如 `${DOUBAO_API_KEY}`。
- 配置加载代码必须支持读取 `.env`，并在缺失环境变量时抛出清晰错误信息。
- `.env` 必须加入 `.gitignore`，避免敏感信息进入版本控制。

### 2. 命名与代码风格

- Python 模块文件统一使用 `snake_case`，不要混用 `Workflow.py` 这类 PascalCase 文件名。
- 包的 `__init__.py` 保持轻量，避免为导出单个类而引入重依赖链。
- 保持命名风格一致：类名使用 PascalCase，函数、变量、模块使用 snake_case。
- 删除未使用的导入和无效兼容代码，避免 reviewer 一眼看到低质量痕迹。

### 4. 文档与实现一致性

- README、QUICKSTART、验证报告中的 Python 版本、测试数量、启动命令、配置方式必须和当前代码一致。
- 文档中不要再要求用户把真实密钥直接写进 `config.yaml`。
- `config.yaml` 仍然是项目规定的配置入口；如果密钥通过环境变量注入，也要通过 `config.yaml` 中的占位符引用，不要绕开它单独搞另一套配置体系。
- 运行说明必须给出最短可执行路径
- 除了 README 其他文档都应该放到 `docs/` 目录下。

### 5. 测试覆盖要求

- 除了业务功能测试，还要覆盖配置链路测试。
- 至少补充以下测试：
- `.env` 占位符解析成功
- 缺失环境变量时报错
- Workflow 主流程可初始化
- RAG 本地文件存储与检索流程
- 自定义工具 `ItineraryPlanner` 和 `FlightBookingSystem`
- 改完后必须重新跑完整测试，并以实际结果更新文档中的测试数量描述。

### 6. 交付完整性检查

- 最终交付前要自查是否包含以下核心文件：
- `main.py`
- `requirements.txt`
- `config.yaml`
- `.env.example`
- `README.md`
- `tests/`
- 如果项目要求 Workflow / RAG / Tools / Memory，则对应源码文件必须完整存在，不能只在文档里声明。

### 7. 推荐的交付顺序

1. 先实现核心功能。
2. 再补测试，确保所有功能可验证。
3. 再补 `.env.example`、README。
4. 最后跑完整测试和启动验证，并同步更新文档。

#### 8.4 常见安全问题规避

- **SQL 注入**：使用参数化查询，不要拼接 SQL 字符串
- **命令注入**：避免使用 `os.system()`、`subprocess.shell=True`
- **路径遍历**：验证用户输入的文件路径，使用 `Path.resolve()` 规范化
- **敏感信息泄露**：
  - 不在日志中输出完整的 API Key
  - 异常信息中不包含敏感配置
  - 使用 `***` 或截断方式显示密钥（如 `sk-***abc`）
